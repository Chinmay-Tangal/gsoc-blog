Large Scale HEOM Solver Using PETSc - GSoC 2026

Student: Chinmay Tangal Organization: QuTiP - Quantum Toolbox in Python, NumFOCUS
Mentors: Alex Pitchford, Emil VATAI, Neill Lambert
Project Duration: May 2026 - August 2026

Project Overview

The Problem

HEOM — the Hierarchical Equations of Motion — is one of the few methods that can exactly simulate open quantum system dynamics without Markov or perturbative approximations. It does this by propagating an entire hierarchy of coupled auxiliary density matrices simultaneously, which is what makes it exact and also what makes it expensive. Memory and compute blow up fast as hierarchy depth or system size grows.

The current QuTiP implementation runs on a single node. The entire hierarchy state lives in one vector on one machine, integrated with a single scipy ODE call. For small systems that’s fine. Push the hierarchy depth past 6, or work with anything beyond a few qubits, and you hit a memory cliff with no workaround — it’s a structural limitation.

What This Project Builds

A distributed execution path for the HEOM solver using PETSc and MPI, so the ODE integration can run across multiple nodes on an HPC cluster. The existing solver stays completely untouched — users who don’t need distributed execution shouldn’t have to think about any of this.

Key Goals

  • Distributed data layout for the HEOM hierarchy across MPI ranks, designed to keep inter-rank communication local
  • PETSc TS integration for distributed ODE integration across ranks
  • Correctness first — the distributed path must produce identical results to the existing solver before any performance work
  • Benchmarks across hierarchy depths, system sizes, and node counts
  • HPC documentation for running the solver on real cluster infrastructure

Implementation

The Design Problem

The coupling structure in HEOM means you can’t naively slice the hierarchy across MPI ranks — adjacent levels need to communicate, so the partition strategy directly determines whether you get a speedup or just move the bottleneck. The key question is whether the data can be laid out so that most communication is between neighboring ranks rather than all-to-all.

Tech Stack

  • PETSc — distributed vectors, matrices, and ODE integration via the TS interface
  • MPI / mpi4py — inter-process communication
  • petsc4py — Python bindings for PETSc
  • QuTiP — existing HEOM infrastructure, benchmark systems, test suite

What Stays Untouched

The existing HEOMSolver is completely unmodified. The distributed path is additive — users running small systems on a single machine see no changes.
